# Research pass: pitfalls for v1.1 `reinstall`

Overall confidence: **High**. Evidence is from current milestone requirements plus install/uninstall/update rollback, cleanup, and concurrency tests.

## Critical pitfalls

| Pitfall | Prevention strategy | Phase to address | Test implications | Confidence |
|---|---|---|---|---|
| Implementing reinstall as `uninstall` then `install` violates the core requirement: uninstall deletes resources/state before install and `installPlugin` refuses already-installed records. Existing uninstall removes the state record before post-cleanup (`uninstall.ts:101-135,167-184`); install fails when record exists (`install.ts:191-198`). | Build a dedicated atomic reinstall flow. It must prepare new resources before any old-resource removal and must not remove the state record as an intermediate observable state. | Core orchestrator design/implementation | Add failure tests proving old state + old skill/command/agent/MCP resources remain after prepare/commit failure. | High |
| Copying `update` is not sufficient for atomic preservation. `update` swaps state before physical replace (`update.ts:497-538`) and then commits bridge writes (`update.ts:590-666`); on phase-3 failure it records failure + recovery hint, not rollback to the previous install. | Treat update as a useful preflight/target-enumeration reference, not the atomicity model. Reinstall needs either reversible per-bridge swaps/backups or another proven preservation mechanism. | Core orchestrator design | Force commit-time filesystem collision like `update.test.ts:666-717`; assert old resources still present and state still references old resources/version. | High |
| Bridge "restage" commits delete previous resources before renaming staged replacements. Passing `previousSkillNames`/`previousCommandNames` into existing commit helpers can remove old resources before a later failure. Agents commit also removes previous owned files. | Audit each bridge commit path before reuse. If using existing prepare/commit helpers, wrap them in a restore-capable transaction or create reinstall-specific safe swap primitives. | Bridge/orchestrator integration | Per bridge: simulate failure after old removal and assert old content restored/preserved. Include agents-index and `mcp.json`. | High |
| Violating no-network reinstall policy by copying `updatePlugins` sync behavior. v1.1 requires cached manifests + no network (`PROJECT.md:23,44`), while update imports/defaults git ops and refreshes GitHub clones (`update.ts:147-193`). | Reinstall orchestrator must not import `DEFAULT_GIT_OPS`, `refreshGitHubClone`, `platform/git`, or expose `gitOps`; read `mp.manifestPath` only. | Core orchestrator + architecture tests | Add source-grep test like install/uninstall NFR-5 tests (`install.test.ts:1061-1071`, `uninstall.test.ts:675-685`). | High |
| Recomputing/bumping the plugin version during reinstall. Requirement says reuse recorded versions (`PROJECT.md:23,44`); install/update use `resolvePluginVersion` from manifest/hash (`install.ts:263-264`, `update.ts:374-377`). | Use the installed state record's `version` for the replacement record. Do not treat manifest `entry.version`/hash as the target version unless requirements explicitly change. | Requirements/design + core orchestrator | Test manifest version differs from installed record; reinstall success must keep recorded `version`. | High |
| Deleting plugin data dir too early. Requirement says delete plugin data directories only after successful replacement (`PROJECT.md:25,46`). Existing uninstall deletes data dir only post-state-commit and downgrades failures to warning (`uninstall.ts:167-184`). | Move data-dir `rm -rf` strictly after successful resource + state replacement. Treat cleanup failure as warning, not operation failure; consider whether to recreate data dir afterward to match install behavior. | Core orchestrator cleanup path | Test data dir survives failed reinstall and is removed only after success; chmod failure should warn and keep successful reinstall state. | High |
| Concurrency drift/orphans if reinstall does long preflight outside lock without stale checks. `withStateGuard` is the cross-process lock and lock losers surface `STATE_LOCK_HELD_PREFIX` (`with-state-guard.ts:55-100`, `markers.ts:37`). Concurrent install tests assert one winner/no orphan skills (`concurrent-install.test.ts:178-237`). | Decide and document lock boundary. At minimum re-read inside `withStateGuard` and verify installed record/version/resources still match the preflight snapshot before replacing. Prefer holding the lock across critical disk/state mutation. | Core orchestrator + integration | Add same-plugin and different-plugin reinstall races; assert one clean winner or serial success, no orphan resources, loser message uses lock marker if applicable. | High |

## Moderate pitfalls

| Pitfall | Prevention strategy | Phase to address | Test implications | Confidence |
|---|---|---|---|---|
| Self-conflict in cross-plugin name guard. Update removes the plugin's own record before conflict checks (`update.ts:560-568`) because otherwise unchanged names collide with itself. | Reinstall must exclude the target plugin's current resources when checking generated skill/command/agent names; still detect conflicts with other plugins in same scope. | Core preflight | Test reinstall of same generated names succeeds; reinstall that collides with another plugin fails before mutation. | High |
| Mishandling missing/non-installable manifest entries. Update partitions these as skipped (`update.ts:339-371`); install hard-fails. Reinstall requirements imply installed targets only, cached manifest read, preserve old on failure. | Choose explicit direct/batch behavior: skip vs error. In all cases, do not remove old install when manifest entry is absent or no longer installable. | Requirements/design | Tests for entry missing, schema invalid, no longer installable: old resources/state/data dir remain. | Medium-High |
| Batch semantics left ambiguous. Update enumerates bare/`@mp`/plugin targets and partitions results (`update.ts:147-232`, `update.ts:704-720`); reinstall requires analogous forms but not explicit continue-vs-abort behavior. | Decide in planning: per-plugin atomicity with partitioned continuation is safest. Avoid one plugin failure aborting after partial work without clear output. | Requirements/design + orchestrator | Tests for bare and `@mp`: one failing plugin does not corrupt others; output names successes/failures deterministically. | Medium |
| Edge/router/completion omissions. Current router and completions only know install/uninstall/update/list/marketplace (`router.ts:33-55,97-104`; `provider.ts:39-48,207-218`). | Add `reinstall` everywhere `update` is accepted: router usage, handler, register deps, completions. Completion mode should filter installed plugins and allow `@marketplace`. | Edge integration | Router usage tests, handler parse tests for bare/`@mp`/`plugin@mp`, `--scope` any position, completion tests for installed-only + marketplace-only. | High |
| Reload-hint verb drift. `reloadHint` only supports `load|refresh|drop` (`reload-hint.ts:15-39`); reinstall should likely use `refresh` for replaced resources and suppress when no generated resources changed. | Do not add a new verb unless requirements and marker tests are updated. Use centralized `reloadHint`/`appendReloadHint`. | Presentation/output | Tests for one/many/no resources: `Run /reload to refresh it.` or suppressed for empty plugin. | High |
| Soft-dependency warnings may be forgotten. Existing install/update/uninstall append pi-subagents/pi-mcp-adapter warnings based on staged/dropped agents/MCP resources (`install.ts:555-596`, `uninstall.ts:209-227`, `update.ts:704-720`). | Reinstall success output should probe staged new agents/MCP servers and append stable warnings from existing helpers. | Presentation/output | Tests with no `subagent`/`mcp` tools and plugins containing agents/MCP. | High |
| Agent foreign-content behavior is easy to get wrong. Install preserves foreign rows as warning (`install.ts:377-385`); uninstall treats foreign content as failure and retains state (`uninstall.test.ts:316-393`). | Decide reinstall semantics before coding. Atomic-preservation requirement suggests foreign previous agent content should block replacement or be preserved without deleting old state/resources. | Requirements/design + bridge integration | Tests with agents-index row pointing to foreign target; assert no deletion and old install preserved or warning behavior matches decision. | Medium-High |
| Completion cache invalidation may be skipped because status remains "installed". Install/uninstall/update all drop marketplace cache post-success (`install.ts:542-552`, `uninstall.ts:154-164`, `update.ts:669-697`). | Invalidate target marketplace cache after successful reinstall; cleanup failure should warn, not fail. | Core post-success | Cache prewarm test analogous to install/uninstall invalidation tests. | Medium |
| Error-surface regression. `PathContainmentError` must bypass rollback-partial markers (`rollback.ts:31-49`), stable marker strings live in `markers.ts:9-37`, and failures should go through `notifyError`/`notifyWarning`, not stdout. | Reuse `formatRollbackError`/`formatErrorWithCauses` or preserve equivalent behavior in custom transaction code. Do not inline stable prefixes. | Core error handling + review | Tests for symlink/path containment: message has original containment text and no `(rollback partial:` unless actual undo partials apply. | High |

## Minor pitfalls

| Pitfall | Prevention strategy | Phase to address | Test implications | Confidence |
|---|---|---|---|---|
| Forgetting two-scope semantics/ambiguity. Update resolves explicit marketplace targets via `resolveScopeFromState` when `--scope` is omitted and bare form enumerates selected scopes (`update.ts:731-780`). | Mirror update's target enumeration exactly unless requirements say otherwise: bare without scope = both scopes; `@mp`/`plugin@mp` without scope may be ambiguous. | Edge/orchestrator | Tests for user+project installed plugins, explicit `--scope`, ambiguous marketplace error. | High |
| Unstable or noisy success output. Existing update uses deterministic partition rendering and sorted updated names (`update.ts:704-720`); reload hints quote multiple names via centralized composer. | Sort names in batch output and keep messages concise. | Presentation/output | Snapshot-ish assertions for ordering of Updated/Reinstalled/Skipped/Failed sections. | Medium |
| Direct `process.cwd()` in helper APIs. `updateSinglePlugin` cascade defaults to `process.cwd()` (`update.ts:247-253`), which is awkward in tests and can drift from ctx.cwd. | Prefer direct reinstall API accepting `cwd`; if adding cascade/internal helper, provide cwd injection. | API design | Avoid tests that must `process.chdir` except for legacy cascade compatibility. | Medium |
| Over-broad MCP conflict assumptions. Orchestrator cross-plugin guard excludes MCP (`plugin/shared.ts:26-36,130-132`); MCP collision belongs to bridge layer. | Do not add ad-hoc MCP name checks in reinstall orchestrator; let `prepareStageMcpServers` enforce its contract. | Core implementation | MCP conflict test should expect bridge error and old install preserved. | Medium |
| Editing stable marker constants unnecessarily. Existing markers are drift-guarded (`markers.ts:1-13,15-37`). | Only add/change marker constants with snapshot/docs updates and a clear requirement. | Review | Run marker snapshot tests if marker surface changes. | High |
