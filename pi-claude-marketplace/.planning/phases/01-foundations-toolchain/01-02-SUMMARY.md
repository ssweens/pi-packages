## Decisions Made

- **Drop the inline `eslint-disable-next-line` belt-and-suspenders comments in notify.ts.** The plan recommended keeping them as redundancy with BLOCK B's per-file override. In practice, with the override turning `no-restricted-syntax` off for the entire file, every inline disable becomes an *unused* directive and triggers `reportUnusedDisableDirectives` warnings, which break `npm run lint`. The cleaner shape: the BLOCK B override is the single audit surface, the file-level docstring documents the sanctioned-use intent, and the function bodies stay clean. PATTERNS.md acknowledged this trade-off ("the planner picks one"); I picked the cleaner one.
- **Route `notifyError`'s optional unknown cause through `errorMessage()` from shared/errors.ts** rather than inline `cause instanceof Error ? cause.message : String(cause)`. The inline form triggers `@typescript-eslint/no-base-to-string` when nested inside a template-literal interpolation (e.g., `` `\nCause: ${... String(cause)}` ``); calling the shared primitive doesn't. Net wire output is identical.
- **Add `@types/write-file-atomic@^4.0.3`** as a dev dep (Rule 3 auto-fix). write-file-atomic ships JS-only; without DefinitelyTyped types the strict-mode typecheck flags `Could not find a declaration file for module 'write-file-atomic'` and the `default import` becomes `any`, cascading into `@typescript-eslint/no-unsafe-call` errors.
- **Trailing whitespace on the first three markers preserved verbatim** (`"pi-subagents is not loaded; "`, `"pi-mcp-adapter is not loaded; "`, `"Run /reload to "`). The trailing space is intentional -- runtime concatenation appends without re-introducing spacing. Future Plan 05 PRD-snapshot test must compare byte-for-byte.

**2. [Rule 1 - Bug] notifyError @typescript-eslint/no-base-to-string error**

- **Found during:** Task 2 (`npm run lint` after first write of shared/notify.ts)
- **Issue:** The plan body had `\`\\nCause: ${cause instanceof Error ? cause.message : String(cause)}\``. ESLint flagged the `String(cause)` inside the template-literal interpolation as `'cause' may use Object's default stringification format ('[object Object]') when stringified` (`@typescript-eslint/no-base-to-string`). The same `String(err)` form passes lint inside `errors.ts` because it's not nested in a template-literal interpolation.
- **Fix:** Imported `errorMessage` from shared/errors.ts and used `errorMessage(cause)` -- the shared primitive normalizes Error/non-Error narrowing in one place, and the lint rule does not flag the call. Wire-level output is identical to the original (`Error.message` for Error instances, `String(value)` for everything else).
- **Files modified:** `extensions/pi-claude-marketplace/shared/notify.ts`
- **Verification:** Runtime smoke (4 wrappers x mock ctx) records the expected `[["a"],["b","warning"],["c\\nCause: boom","error"],["bad args\\n\\nUsage: /claude:plugin install <plugin>@<marketplace>","error"]]` exactly. `npm run lint` passes.
- **Committed in:** `e2fd132` (Task 2 commit, in the initial Task 2 file body)
