# Phase 3: Resource Bridges -- Research

**Researched:** 2026-05-10
**Domain:** Atomic per-bridge staging (skills/commands/agents/MCP) translating Claude plugin artefacts into Pi-native artefacts
**Confidence:** HIGH (V1 implementation present, exhaustively read; Phase 1/2 carry-forward verified against on-disk source)

This research file is **prescriptive, not exploratory**. CONTEXT.md locked 10 decisions (D-01..D-10). Phase 3 owns 33 v1 REQ-IDs. The job here is to surface the concrete bridge interface contracts, the per-bridge prepare/commit/abort algorithms, the marker/ownership/collision predicates, and the atomicity story -- with enough specificity that the planner can write executable plan files without re-reading V1.

The four bridges differ materially in IO shape (directory copy / per-file rename / JSON merge), so the unified surface lives at the **per-bridge function signature level** -- not at a single `Bridge<P>` interface. Per CONTEXT.md D-01 there is no shared interface; each bridge exports concrete `prepare`/`commit`/`abort`/`unstage` functions taking a typed input and returning a discriminated `Prepared<bridge>` opaque handle.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01 (Per-bridge concrete signatures with opaque `Prepared<bridge>` handles):** No uniform `Bridge<P>` interface. Each bridge owns its own typed `prepare(input): Promise<Prepared<bridge>>`, `commit(prep): Promise<CommitResult>`, `abort(prep): Promise<void>`, `unstage(target): Promise<UnstageResult>`. The unified composition surface lives at Phase 5's literal-array `Phase<C>` ledger, not in `bridges/`.
- **D-01 corollary:** `bridges/` may NOT import from `transaction/` (Phase 1 D-11). Bridges are ledger-agnostic primitives.
- **D-02 (bridge-as-Phase composition; documented for Phase 5):** Phase 3 ships bridges with `prepare/commit/abort/unstage` as raw functions and unit-tests each in isolation. Phase 5's `orchestrators/install.ts` (and `update.ts`, `uninstall.ts`) is the only consumer that composes them into a `runPhases<InstallCtx>` ledger. The composition pattern is bridge-as-Phase: each bridge maps to one literal `Phase<C>` entry where `do = bridge.commit(prep)` and `undo = bridge.unstage(target)`.
- **D-03 (Agent conversion in `bridges/agents/`):** Frontmatter parse + emit (AG-6, AG-8) and AG-7 field mapping live in `bridges/agents/{frontmatter,convert}.ts`. NOT in `domain/`. Mirrors V1's `agent/{convert,frontmatter}.ts` split.
- **D-04 (Compute-target-then-atomic-apply per bridge):** Each bridge's `commit` uniformly does (1) identify owned artefacts via index/state/marker check; (2) compute target spec = (existing-not-owned ∪ new-staged); (3) atomic-apply. Install/uninstall/update re-stage collapse into the same primitive.
- **D-05 (Per-bridge collision detection; cross-bridge guard is Phase 5):** Each bridge enforces same-bridge collisions and cross-owner refusals during `prepare`. Cross-bridge PI-6 (skill name == command name == agent name) runs in Phase 5's `orchestrators/install.ts`.
- **D-06 (Markers per bridge):** Skills + commands have NO marker (path is the marker). Agents use a TWO-PART marker: (a) basename starts with `pi-claude-marketplace-`, (b) body contains the verbatim `generated by pi-claude-marketplace` string in an HTML-comment provenance block. MCP uses a `_piClaudeMarketplace: { plugin, marketplace }` field in each staged server entry.
- **D-07 (`agents-index.json` schema in `persistence/`):** TypeBox schema + JIT validator in `persistence/agents-index-schema.ts`. IO-side reader/writer (`loadAgentsIndex`, `saveAgentsIndex`) in `persistence/agents-index-io.ts`. Bridge-level mutation logic in `bridges/agents/index.ts`.
- **D-08 (`shared/vars.ts` for `${CLAUDE_PLUGIN_ROOT}` / `${CLAUDE_PLUGIN_DATA}`):** Pure function `substituteClaudeVars(body, vars)`. Skills + commands consume; agents do NOT (no AG-* requirement).
- **D-09 (Reuse Phase 2 `domain/name.ts`):** Phase 3 verifies SK-2 + CM-2 elision logic exists in Phase 2's `domain/name.ts` and adds tests if missing.
- **D-10 (Phase 3 ships `resources_discover` per-scope helper only):** `bridges/skills/discover.ts` returns discovered skill directories from one scope + per-scope error. Phase 7's `index.ts` does the actual `pi.on('resources_discover', ...)` wiring + per-scope error aggregation per SK-5.

## Project Constraints (from CLAUDE.md)

- **NFR-1:** All disk mutations atomic (tmp + rename or atomic JSON write). Phase 3 uses `atomicWriteJson` from Phase 1 for `mcp.json` + `agents-index.json`; per-file `rename` for staged files; per-tree `rename` for skill directories.
- **NFR-3:** All operations safe to retry -- idempotent or fail-clean. Bridge `prepare/commit/abort/unstage` MUST be idempotent on retry (e.g., re-running `unstage` for a plugin whose entries already gone returns `{removedNames: []}`, not an error).
- **NFR-4:** Node ≥ 22. Use `node:fs/promises`, `node:crypto::randomUUID`, `node:path`, `node:os` only.
- **NFR-7:** Bridges read `pluginRoot` only from `ResolvedPlugin & { installable: true }` variant; the type discriminator already enforces this.
- **NFR-10:** Refuse to write outside `<scopeRoot>/pi-claude-marketplace/`, `<scopeRoot>/agents/`, or `<scopeRoot>/mcp.json`. Every name-derived path goes through `assertPathInside` (Phase 1 chokepoint).
- **NFR-5:** Network forbidden in install/uninstall flow. Bridges MUST NOT touch network -- all paths derived from `ScopedLocations` + plugin source dir.
- **IL-2:** Every user-visible message via `ctx.ui.notify` -- **bridges do NOT route through notify directly.** Bridges throw or return `CommitResult` containing `warnings[]` / `failed[]`; Phase 4/5 orchestrators surface those via `notifyWarning` / `notifyError`.
- **TypeScript strict.** Discriminated unions throughout. No `any`, no implicit `unknown` propagation.

## Executive Summary

Phase 3 ports the four V1 staging modules (`agent/stage.ts`, `mcp/stage.ts`, `resource/stage.ts`, plus the `agent/{convert,frontmatter}.ts` and `mcp/{parse,marker,effective-config}.ts` helpers) into the new 9-folder layout's `bridges/{skills,commands,agents,mcp}/` directories, refactoring V1's combined `resource/stage.ts` into separate `bridges/skills/` and `bridges/commands/` so each bridge ships its own `prepare/commit/abort/unstage` quartet.

The bridges are **ledger-agnostic primitives**: each exports concrete typed functions (no shared `Bridge<P>` interface per D-01) and receives the inputs Phase 5's orchestrator computes. V1's bridge surfaces are sound and carry forward almost verbatim with three deltas:

1. **Module split.** V1 collapses skills + commands into one `resource/stage.ts`; Phase 3 splits them into `bridges/skills/` and `bridges/commands/` so each owns its own `prepare/commit/abort/unstage` files. The discovery step (currently in V1 `discoverPluginResources`) splits accordingly: skills discovery to `bridges/skills/discover.ts` (also serving SK-5), commands discovery to `bridges/commands/discover.ts`.
2. **Bridge-local prepare/commit/abort.** V1 has prepare/commit for agents and MCP but not for skills/commands (skills/commands stage directly via `stagePluginResources` into a tmp tree owned by the orchestrator). Phase 3 adds explicit `prepareStageSkills` / `prepareStageCommands` returning discriminated `Prepared<bridge>` handles, so all four bridges expose the same primitive shape and Phase 5's ledger composition is uniform.
3. **`agents-index.json` schema lives in `persistence/`.** V1 mixes the index schema validation into the bridge code (`validateAgentIndexEntry` inside `agent/stage.ts`). Phase 3 hoists the schema + JIT validator into `persistence/agents-index-schema.ts` and the IO into `persistence/agents-index-io.ts` per D-07, so a future inspection tool can read the index without depending on `bridges/agents/`.

**Primary recommendation:** Carry V1 forward. The implementation is correct -- the 33 REQ-IDs map 1:1 to V1 code paths that already pass tests. Phase 3's risk is shape (where files live, how the discriminated handles type-check across per-bridge consumers) and the addition of the prepare/commit split for skills/commands -- not algorithmic novelty.

## V1 Baseline (Per Bridge)

### V1: SkillsBridge baseline

**V1 source:** `extensions/pi-claude-marketplace/resource/stage.ts` (combined skills + commands).

**What V1 does:**
- `discoverPluginResources(resolved: ResolvedPluginInstallable): Promise<DiscoveredResources>` reads `componentPaths.skills` (via `resolveComponentDirectory`), enumerates direct subdirectories of the skills component dir, filters those containing a `SKILL.md` file, and returns a `DiscoveredSkill[]` with `{name, generatedName, skillDir}`.
- `stagePluginResources(input)` is a single-phase stage (no separate prepare/commit). For each skill: `cp -r` from `skill.skillDir` to `<stagingDir>/skills/<generatedName>/`, then `readFile` the staged `SKILL.md`, rewrite the frontmatter `name:` field via `rewriteFrontmatterName`, run `substitutePluginVars` over the body, write back.
- The orchestrator (`plugin/install.ts`) owns the staging tmp dir; this function writes INTO that tmp dir and the orchestrator `rename`s the whole tree to its final destination at commit.

**What V1 gets right (carry forward unchanged):**
- Directory copy via `fs.cp({recursive: true})` -- correct for full skill content (resources, scripts, etc. that may sit alongside `SKILL.md`).
- Skip skill subdirs missing `SKILL.md` -- avoids treating a stray `__pycache__/` as a skill.
- Variable substitution applied only to `SKILL.md`, not to other files in the skill dir -- correct (PRD §5.5 mentions `SKILL.md` specifically).
- Frontmatter rewrite preserves all other fields (`description`, `license`, etc.).

**What V1 lacks that this phase must add:**
- **Bridge-local prepare/commit/abort.** V1 inlines staging into the orchestrator's tmp dir. Phase 3 needs `prepareStageSkills` returning a `PreparedSkillsStaging` handle (its own UUID staging dir under `<extensionRoot>/skills-staging/<uuid>/`), `commitPreparedSkills` doing the per-skill atomic dir rename into `<scope>/pi-claude-marketplace/resources/skills/<generatedName>/`, and `abortPreparedSkills` cleaning up the staging dir.
- **Per-skill atomic apply at commit, not orchestrator-level tree rename.** V1's "commit the whole tree" pattern is incompatible with the bridge-as-Phase model (D-02): each bridge owns its own atomic boundary so re-staging a single bridge doesn't disturb the others.
- **`bridges/skills/discover.ts`** -- a per-scope `resources_discover` helper for SK-5 / D-10. V1 has no equivalent (V1 emits `resources_discover` from `index.ts`); Phase 3 ships the helper; Phase 7 wires the event.
- **RN-6 collision check** -- two skill source names that elide to the same generated name MUST throw with both source names listed. V1 doesn't currently throw (fortunately rare in practice but spec-required).

**V1 control flow (install path):**
```
plugin/install.ts
  -> discoverPluginResources(resolved)             # reads from pluginRoot
  -> stagePluginResources({skills,commands,...,stagingDir})  # writes to <stagingDir>/skills/...
  -> rename(stagingDir, finalDestDir)              # orchestrator-level commit
```

- For each: writes `<stagingDir>/prompts/<generatedName>.md` with substituted body.

### V1: AgentsBridge baseline

**V1 source:** `extensions/pi-claude-marketplace/agent/{stage,convert,frontmatter}.ts` -- 866 lines of mature, well-tested logic.

**What V1 does:**
- `discoverPluginAgents` reads `agentsDir`, filters `.md` files non-recursively, parses each via `parseFrontmatter`, computes `sourceHash` (sha256 over raw bytes -- BOM/line-ending tolerant), produces `DiscoveredAgent[]`.
- `convertAgent` performs the AG-7 mapping pipeline: description fallback → model mapping (`MODEL_MAP`) → tools mapping (`TOOL_MAP`) with `disallowedTools` filtering → thinking/effort fallback → skills resolution against `knownSkills` → dropped-fields collection → variable substitution in body → `emitGeneratedAgentFile` assembly.
- `prepareStagePluginAgents` returns `PreparedAgentsStaging = PreparedAgentsNoop | PreparedAgentsStaged`. Steps: discover → collision-detect (AG-12) → load index → partition by `(marketplace, plugin)` → cross-owner guard (AG-9) → noop branch (AS-9) → safety-check previous targets (foreign content refusal) → write staged files into `<extensionRoot>/agents-staging/<uuid>/` → return staged paths.
- `commitPreparedAgents` removes previous target files (ENOENT-tolerant) → renames staged files into `<scope>/agents/` → `saveAgentIndex(merged)` → cleanup staging.
- `abortPreparedAgents` cleans up the staging dir (no-op for noop branch).
- `unstagePluginAgents` loads index → filters by `(marketplace, plugin)` → safety-checks each target → `rm` each → removes matching index rows → `saveAgentIndex`.

**What V1 gets right (carry forward UNCHANGED):**
- Two-phase prepare/commit with discriminated `noop | staged` union.
- Cross-owner ownership guard with formatted `"<name>" already owned by <other-mp>/<other-plugin>"` error message.
- Foreign-content refusal at prepare time (basename + marker check).
- Partitioning the index by `(marketplace, plugin)` tuple -- only owning rows touched on re-stage.
- `loadAgentIndex` distinguishes file-level corruption (throw) from per-row corruption (drop row + collect into `corruptions[]` returned through prepared.result.warnings).
- Per-file `rename` granularity -- partial commit failures self-heal on retry because `unstage` tolerates ENOENT on previous targetPaths.
- `MODEL_MAP`, `TOOL_MAP`, `THINKING_VALUES` locked as user contract -- a downstream user is depending on `sonnet -> anthropic/claude-sonnet-4-6`, etc.
- Provenance HTML-comment block placed AFTER the closing `---` (so pi-subagents' frontmatter parser still sees the frontmatter as the file's first thing).
- `emitYamlScalar` quote-flip + `sanitizeProvenance` `-->` escape -- both belong to the AG-8 parser-safety guarantee.
- `dedupePreservingOrder` for tools list -- correct behavior when source has duplicates.

**What V1 lacks that this phase must add:**
- **Schema in `persistence/agents-index-schema.ts` (D-07).** V1 has hand-rolled `validateAgentIndexEntry` inside `agent/stage.ts`. Phase 3 hoists this to a TypeBox JIT-compiled validator alongside Phase 2's `state-schema.ts` so a future inspection tool can validate the index without depending on bridges.
- **`persistence/agents-index-io.ts` separation.** V1's `loadAgentIndex` / `saveAgentIndex` live in `agent/stage.ts`. Phase 3 hoists the IO layer to `persistence/` per the 9-folder layout; the bridge-side mutation logic (`partitionByOwner`, `mergeWithPrevious`) stays in `bridges/agents/index.ts`.
- **`bridges/agents/marker.ts`** for the two-part check predicate. V1 has `isSafeToTouch` inside `agent/stage.ts`; Phase 3 hoists to a named module so the predicate is testable in isolation.
- **`AgentForeignContentError`** typed error per CONTEXT.md "Existing Code Insights." V1 throws plain `Error("Refusing to overwrite agent file at ${targetPath}: ${reason}")` -- Phase 3 makes it a typed subclass of `PathContainmentError` (per Phase 1 D-17 inheritance pattern).

**V1 control flow (install path):**
```
prepareStagePluginAgents(input)
  -> discover + collision-check + load-index + partition + cross-owner-guard
  -> noop branch (return early) OR
  -> safety-check previous + write to staging tmp + compute newEntries

commitPreparedAgents(prepared)
  -> rm previous targets (ENOENT-tolerant)

abortPreparedAgents(prepared)
  -> cleanup staging (no-op for noop)
unstagePluginAgents(input)
```

### V1: McpBridge baseline

The `_`-prefixed fields are opaque to consumers but TypeScript-visible (consumers can `commit(prepared)` but the `commit` implementation reads them).
```typescript
// PreparedSkillsStaging:    kind: "noop" | "staged"             // local to skills
// PreparedCommandsStaging:  kind: "noop" | "staged"             // local to commands
// PreparedAgentsStaging:    kind: "noop" | "staged"             // local to agents
// PreparedMcpStaging:       kind: "noop" | "staged"             // local to mcp
```

The shared kind names (`"noop"` / `"staged"`) are NOT a problem because the bridges' `commit*` functions only accept their OWN `Prepared<bridge>` type by name -- the `_marketplaceName`, `_newRenamePairs`, `_nextDoc`, `_stagedFilePaths` fields differ per bridge, so structurally a mis-typed pass would fail.

**For absolute belt-and-braces type-distinct handles**, add a brand symbol per bridge (D-01 corollary):

```typescript
  // ... rest
```

### `prepareStageSkills(input): Promise<PreparedSkillsStaging>`

```typescript
export async function prepareStageSkills(
  input: StageSkillsInput
): Promise<PreparedSkillsStaging> {
  const { locations, marketplaceName, pluginName, pluginRoot, pluginDataDir, resolved } = input;

  // 1. Discover.
  const discovered = await discoverPluginSkills({ pluginName, resolved });

  // 2. RN-6 collision check: two source names eliding to same generated name.
  assertNoSkillCollisions(discovered);  // throws with both source names listed

  // 3. Load previous-ours from state. Phase 5 passes previousSkillNames in input.
  //    For Phase 3 unit tests, accept `previousNames: readonly string[]` directly.
  const previousNames = input.previousSkillNames ?? [];

  // 4. Noop branch (does NOT trigger from RN-6 specifically; from "no inputs at all").
  //    For skills, "noop" means: no discovered AND no previous to drop.
  //    This bridge does NOT have a materialization gate equivalent to AS-9 (skills dir
  //    always exists once anything is in it; no dedicated gate REQ-ID for skills).
  //    But for the prepare/commit symmetry we still avoid creating staging dirs when
  //    there's no work to do.
  if (discovered.length === 0 && previousNames.length === 0) {
    return { kind: "noop", result: { stagedNames: [], warnings: [] } };
  }

  // 5. Cross-marketplace conflict: a skill dir already exists at target with a
  //    different (mp, plugin)? Skills have no marker -- the dir name is the marker.
  //    Phase 5 must pre-check via state.json (PI-6 cross-plugin guard); Phase 3
  //    bridge does NOT cross-check across owners (per D-05). Skipped here.

  // 6. Stage. Per-skill: cp -r into <extensionRoot>/skills-staging/<uuid>/<generatedName>/,
  //    then rewrite SKILL.md frontmatter + substitute vars.
  const stagingRoot = path.join(locations.extensionRoot, "skills-staging", randomUUID());
  await mkdir(stagingRoot, { recursive: true });

  const newRenamePairs: { from: string; to: string }[] = [];
  const skillsTargetRoot = path.join(locations.extensionRoot, "resources", "skills");

  try {
    for (const skill of discovered) {
      const stagedDir = path.join(stagingRoot, skill.generatedName);
      const targetDir = path.join(skillsTargetRoot, skill.generatedName);
      await assertPathInside(stagingRoot, stagedDir, "staged skill dir");
      await assertPathInside(skillsTargetRoot, targetDir, "target skill dir");

      await cp(skill.skillDir, stagedDir, { recursive: true });

      const skillMdPath = path.join(stagedDir, "SKILL.md");
      let content = await readFile(skillMdPath, "utf8");
      content = rewriteFrontmatterName(content, skill.generatedName);  // SK-3
      content = substituteClaudeVars(content, { pluginRoot, pluginData: pluginDataDir });  // SK-4
      await writeFile(skillMdPath, content, "utf8");

      newRenamePairs.push({ from: stagedDir, to: targetDir });
    }
  } catch (err) {
    throw appendLeakToError(err, await cleanupStaging(stagingRoot, "skills staging directory"));
  }

  return {
    kind: "staged",
    locations,
    stagingRoot,
    result: { stagedNames: discovered.map(s => s.generatedName), warnings: [] },
    _previousNames: previousNames,
    _newRenamePairs: newRenamePairs,
    _marketplaceName: marketplaceName,
    _pluginName: pluginName,
  };
}
```

**Inputs:** `locations` (Phase 2 brand), `pluginName`, `pluginRoot`, `pluginDataDir`, `resolved` (for componentPaths), `previousSkillNames` (read from state by Phase 5 caller -- empty for fresh install, populated for re-stage / update).

**Outputs:** `PreparedSkillsStaging` -- discriminated `noop` / `staged`.

**Errors:** `PathContainmentError` (escape from staging tmp or target), `Error("RN-6 collision: ...")` (two sources elide to same name), `Error` from `cp` / `readFile` / `writeFile`.

**Staging layout:**
```
<extensionRoot>/skills-staging/
  <uuid>/                                            # one staging tree per prepare call
    <plugin>-<skill1>/
      SKILL.md                                       # rewritten + substituted
      <other skill files copied recursively>
    <plugin>-<skill2>/
      ...
```

Same-FS guarantee: `<extensionRoot>/skills-staging/` is on the SAME filesystem as `<extensionRoot>/resources/skills/` (both under `<scopeRoot>/pi-claude-marketplace/`), so the per-skill `rename` at commit is atomic.

### `commitPreparedSkills(prepared): Promise<StageSkillsCommitResult>`

```typescript
export async function commitPreparedSkills(
  prepared: PreparedSkillsStaging
): Promise<StageSkillsCommitResult> {
  if (prepared.kind === "noop") return prepared.result;

  const { locations, stagingRoot, _previousNames, _newRenamePairs } = prepared;
  const skillsTargetRoot = path.join(locations.extensionRoot, "resources", "skills");
  await mkdir(skillsTargetRoot, { recursive: true });

  // 1. Remove previous-ours target dirs (if any). ENOENT-tolerant.
  for (const prevName of _previousNames) {
    const prevDir = path.join(skillsTargetRoot, prevName);
    await assertPathInside(skillsTargetRoot, prevDir, "previous skill dir");
    try {
      await rm(prevDir, { recursive: true, force: true });
    } catch (err) {
      // force:true should swallow ENOENT, but propagate everything else.
      if ((err as NodeJS.ErrnoException).code !== "ENOENT") throw err;
    }
  }

  // 2. Per-skill atomic dir rename. Same-FS, so each rename is atomic.
  //    Run sequentially (Promise.all is safe but sequential makes failure
  //    site easier to reason about; skills counts are small).
  for (const { from, to } of _newRenamePairs) {
    await rename(from, to);
  }

  // 3. Cleanup the now-empty staging UUID dir.
  const leak = await cleanupStaging(stagingRoot, "skills staging directory");

  return {
    stagedNames: prepared.result.stagedNames,
    warnings: leak !== undefined ? [leak] : [],
  };
}
```

**Atomicity story:** Per-skill `rename` is atomic per skill. Multi-skill commits are NOT atomic across skills -- if commit fails after skill 2 but before skill 5, skills 1+2 are in place and 3+5 are still in staging. The next call to `unstagePluginSkills` is idempotent (ENOENT-tolerant `rm`), so retry self-heals. This matches V1's per-file rename pattern for agents -- the same trade-off applied to a different bridge.

### `unstagePluginSkills(input): Promise<UnstageSkillsResult>`

```typescript

  locations: ScopedLocations;
  marketplaceName: string;
  pluginName: string;
  previousSkillNames: readonly string[];  // from state -- Phase 5 caller
}

export async function unstagePluginSkills(
  input: UnstagePluginSkillsInput
): Promise<UnstageSkillsResult> {
  const skillsTargetRoot = path.join(input.locations.extensionRoot, "resources", "skills");
  const removed: string[] = [];
  for (const name of input.previousSkillNames) {
    const dir = path.join(skillsTargetRoot, name);
    await assertPathInside(skillsTargetRoot, dir, "skill to unstage");
    try {
      await rm(dir, { recursive: true, force: true });
      removed.push(name);
    } catch (err) {
      if ((err as NodeJS.ErrnoException).code === "ENOENT") {
        removed.push(name);  // already gone -- treat as removed (idempotent)
      } else {
        throw err;
      }
    }
  }
  return { removedNames: removed, warnings: [] };
}
```

Skills have no marker -- ownership is established by the caller's `previousSkillNames` list (read from state). If a foreign skill dir lives at the target path, this function will happily `rm` it (because the caller said "this is mine"). PI-6 cross-plugin name guard (Phase 5) is the safety net: it runs BEFORE `prepare`, so two plugins can never claim the same generated skill name in the first place.

### `bridges/agents/marker.ts` (AG-5 two-part check)

```typescript
import { readFile } from "node:fs/promises";
import path from "node:path";
import { GENERATED_AGENT_MARKER } from "./frontmatter.ts";  // "generated by pi-claude-marketplace"

/** Filename prefix that identifies a generated agent file on disk. */
export const GENERATED_AGENT_PREFIX = "pi-claude-marketplace-";

export type AgentOwnershipResult =
  | { ok: true }                          // safe to touch (file is ours OR doesn't exist)
  | { ok: false; reason: string };        // foreign content -- refuse

/**
 * AG-5 two-part check for whether a target file is "ours":
 *   (a) basename starts with `pi-claude-marketplace-`
 *   (b) body contains the literal `generated by pi-claude-marketplace`
 *       string (inside an HTML-comment provenance block, but we don't parse
 *       structurally -- substring match is sufficient).
 *
 * ENOENT on the target -> ok (nothing to clobber).
 * Either check failing -> NOT ok with a specific reason for the user.
 * Other read errors (EACCES, EISDIR) propagate to caller.
 */
export async function isOwnedAgentFile(targetPath: string): Promise<AgentOwnershipResult> {
  const base = path.basename(targetPath);
  if (!base.startsWith(GENERATED_AGENT_PREFIX)) {
    return {
      ok: false,
      reason: `target filename "${base}" does not start with "${GENERATED_AGENT_PREFIX}"`,
    };
  }

  let contents: string;
  try {
    contents = await readFile(targetPath, "utf8");
  } catch (err) {
    const code = (err as NodeJS.ErrnoException).code;
    if (code === "ENOENT") return { ok: true };  // doesn't exist -- safe to touch
    throw err;  // EACCES, EISDIR, etc. -- propagate
  }

  if (!contents.includes(GENERATED_AGENT_MARKER)) {
    return {
      ok: false,
      reason: `target ${targetPath} is missing the generated marker "${GENERATED_AGENT_MARKER}"`,
    };
  }

  return { ok: true };
}
```

This predicate is the gate for both:
1. `prepare()` -- when re-staging, the previous target file MUST be ours before we rm it.
2. `unstage()` -- when removing, the target file MUST be ours before we rm it.

If the predicate returns `{ok:false, reason}`, the caller treats this as **foreign content** and:
- During `prepare`: throws `AgentForeignContentError` (extends `PathContainmentError`).
- During `unstage`: records the entry in `failed[]` and preserves the index row (so retry surfaces the same diagnostic).

### `bridges/agents/prepare.ts`

Identical algorithm to V1 `prepareStagePluginAgents` (lines 248-368 above), with three deltas:
1. Uses `loadAgentsIndex` from `persistence/agents-index-io.ts` (TypeBox-validated) instead of V1's hand-rolled `loadAgentIndex`.
2. Uses `isOwnedAgentFile` from `bridges/agents/marker.ts` instead of V1's inline `isSafeToTouch`.
3. Throws `AgentForeignContentError` instead of plain `Error` when foreign content blocks overwrite (so Phase 5 can `instanceof` distinguish from other refusals).

```typescript
export async function prepareStagePluginAgents(
  input: StageAgentsInput,
): Promise<PreparedAgentsStaging> {
  const { locations, marketplaceName, pluginName, pluginRoot, pluginDataDir, knownSkills, agentsSourceDir } = input;

  // 1. Discover (AG-6 frontmatter parsing inside).
  const discovered = agentsSourceDir === ""
    ? []
    : await discoverPluginAgents({ pluginName, agentsDir: agentsSourceDir });

  // 2. AG-12: collision detection within this plugin.
  assertNoAgentCollisions(discovered);

  // 3. AG-7: convert each agent. AG-11 throws inside if mapped-tool list is empty.
  const converted: ConvertedAgent[] = discovered.map(d =>
    convertAgent({ pluginName, pluginRoot, pluginDataDir, knownSkills, discovered: d, sourceHash: d.sourceHash })
  );

  // 4. Load + partition index (AG-3).
  const existingIndex = await loadAgentsIndex(locations);
  const previousEntries = existingIndex.entries.filter(
    e => e.marketplace === marketplaceName && e.plugin === pluginName,
  );
  const otherEntries = existingIndex.entries.filter(
    e => !(e.marketplace === marketplaceName && e.plugin === pluginName),
  );

  // 5. AG-9 / RN-4 cross-owner refusal.
  const otherNames = new Map(otherEntries.map(e => [e.generatedName, e]));
  const conflicts = converted
    .map(c => ({ name: c.generatedName, owner: otherNames.get(c.generatedName) }))
    .filter((x): x is { name: string; owner: AgentIndexEntry } => x.owner !== undefined);
  if (conflicts.length > 0) {
    const list = conflicts
      .map(c => `"${c.name}" already owned by ${c.owner.marketplace}/${c.owner.plugin}`)
      .join("; ");
    throw new AgentOwnershipConflictError(
      `Refusing to stage agents for ${marketplaceName}/${pluginName}: ${list}.`,
      conflicts.map(c => ({ name: c.name, marketplace: c.owner.marketplace, plugin: c.owner.plugin })),
    );
  }

  // 6. AS-9 noop branch.
  if (converted.length === 0 && previousEntries.length === 0) {
    return { kind: "noop", result: { stagedNames: [], warnings: [], failed: [] } };
  }

  // 7. AG-5 foreign-content check on previous targets.
  for (const entry of previousEntries) {
    const ownership = await isOwnedAgentFile(entry.targetPath);
    if (!ownership.ok) {
      throw new AgentForeignContentError(
        `Refusing to overwrite agent file at ${entry.targetPath}: ${ownership.reason}`,
        entry.targetPath,
      );
    }
  }

  // 8. Stage to <extensionRoot>/agents-staging/<uuid>/.
  const stagingDir = path.join(locations.agentsStagingDir, randomUUID());
  await mkdir(stagingDir, { recursive: true });

  const stagedFilePaths: { from: string; to: string }[] = [];
  try {
    for (const c of converted) {
      assertSafeName(c.generatedName);
      const targetFile = path.join(locations.agentsDir, c.generatedName + ".md");
      const stagedFile = path.join(stagingDir, c.generatedName + ".md");
      await assertPathInside(locations.agentsDir, targetFile, "agent target path");
      await assertPathInside(stagingDir, stagedFile, "staged agent path");
      await writeFile(stagedFile, c.fileContent, "utf8");
      stagedFilePaths.push({ from: stagedFile, to: targetFile });
    }
  } catch (err) {
    throw appendLeakToError(err, await cleanupStaging(stagingDir, "agents staging directory"));
  }

  const newEntries: AgentIndexEntry[] = converted.map(c => ({
    plugin: pluginName,
    marketplace: marketplaceName,
    sourceAgent: c.sourceName,
    generatedName: c.generatedName,
    sourcePath: c.sourcePath,
    targetPath: path.join(locations.agentsDir, c.generatedName + ".md"),
    sourceHash: c.sourceHash,
    droppedFields: [...c.droppedFields],
    droppedTools: [...c.droppedTools],
    warnings: [...c.warnings],
    ...(c.originalModel !== undefined ? { originalModel: c.originalModel } : {}),
  }));

  const aggregatedWarnings: string[] = [];
  for (const c of converted) aggregatedWarnings.push(...formatAgentWarnings(c));
  for (const corruption of existingIndex.corruptions) {
    aggregatedWarnings.push(`agent index corruption (entry dropped): ${corruption}`);
  }

  return {
    kind: "staged",
    locations,
    stagingDir,
    result: {
      stagedNames: converted.map(c => c.generatedName),
      warnings: aggregatedWarnings,
      failed: [],  // foreign content was already a throw above; no soft failures here
    },
    _previousEntries: previousEntries,
    _otherEntries: otherEntries,
    _newEntries: newEntries,
    _stagedFilePaths: stagedFilePaths,
    _marketplaceName: marketplaceName,
    _pluginName: pluginName,
  };
}
```

```typescript
/**
 * MC-4 / RN-5: the four mcp.json files pi-mcp-adapter consults.

    path.join(home, ".pi", "agent", "mcp.json"),
    path.join(cwd, ".mcp.json"),
    path.join(cwd, ".pi", "mcp.json"),
```

`loadEffectiveServerNames(cwd)` is identical to V1: walk the four slots in order, each path's `mcpServers` map's keys are added to a `Map<serverName, owningPath>` only if not already present (first-declarer wins).

```typescript
export async function loadEffectiveServerNames(cwd: string): Promise<Map<string, string>> {
  const candidates = MCP_COLLISION_SLOTS(cwd);
  const names = new Map<string, string>();
  for (const candidatePath of candidates) {
    let raw: string;
    try { raw = await readFile(candidatePath, "utf8"); }
    catch (err) {
      const code = (err as NodeJS.ErrnoException).code;
      if (code === "ENOENT" || code === "ENOTDIR") continue;
      throw err;  // EACCES propagates
    }
    let parsed: unknown;
    try { parsed = JSON.parse(raw); }
    catch { continue; }  // malformed JSON contributes nothing -- not our problem
    if (typeof parsed !== "object" || parsed === null || Array.isArray(parsed)) continue;
    const servers = (parsed as Record<string, unknown>).mcpServers;
    if (typeof servers !== "object" || servers === null || Array.isArray(servers)) continue;
    for (const name of Object.keys(servers)) {
      if (!names.has(name)) names.set(name, candidatePath);
    }
  }
  return names;
}
```

```typescript
export async function prepareStageMcpServers(input: StageMcpInput): Promise<PreparedMcpStaging> {
  const { locations, marketplaceName, pluginName, servers } = input;

  // 1. Read scope's mcp.json (ENOENT -> {}).
  const doc = await readScopedDoc(locations.mcpJsonPath);
  const existing = getMcpServers(doc);

  // 2. Partition existing into ours-vs-theirs by marker.
  const ours = new Set<string>();
  const theirs: Record<string, unknown> = {};
  for (const [name, value] of Object.entries(existing)) {
    if (isOwnedBy(value, pluginName, marketplaceName)) ours.add(name);
    else theirs[name] = value;
  }

  // 3. MC-4 collision check across all four slots. Self-replace allowed.
  const newNames = Object.keys(servers);
  if (newNames.length > 0) {
    // Phase 5's orchestrator passes cwd in input. For Phase 3 unit tests, accept input.cwd.
    const effective = await loadEffectiveServerNames(input.cwd);
    for (const name of newNames) {
      if (ours.has(name)) continue;  // self-replace within own scope -- OK
      const owningPath = effective.get(name);
      if (owningPath !== undefined && owningPath !== locations.mcpJsonPath) {
        throw new McpServerCollisionError(
          `Refusing to stage MCP servers for ${marketplaceName}/${pluginName}: ` +
          `name "${name}" already exists in ${owningPath}.`,
          name, owningPath,
        );
      }
      if (Object.prototype.hasOwnProperty.call(theirs, name)) {
        throw new McpServerCollisionError(
          `Refusing to stage MCP servers for ${marketplaceName}/${pluginName}: ` +
          `name "${name}" already exists in ${locations.mcpJsonPath}.`,
          name, locations.mcpJsonPath,
        );
      }
    }
  }

  // 4. AS-8 noop branch.
  if (newNames.length === 0 && ours.size === 0) {
    return { kind: "noop" };
  }

  // 5. MC-5 stamp marker on each new entry.
  const marker = buildMarker(pluginName, marketplaceName);
  const stamped: Record<string, unknown> = {};
  for (const [name, entry] of Object.entries(servers)) {
    if (typeof entry !== "object" || entry === null || Array.isArray(entry)) {
      throw new Error(`server "${name}" entry must be an object.`);
    }
    stamped[name] = { ...(entry as Record<string, unknown>), [CLAUDE_MARKETPLACE_MARKER_KEY]: marker };
  }

  // 6. Compute next doc in memory. Removes our-previous (so update doesn't double them);
  //    keeps theirs untouched; adds new stamped.
  const next: RawMcpDoc = { ...doc, mcpServers: { ...theirs, ...stamped } };

  return { kind: "staged", locations, stagedNames: newNames, _nextDoc: next };
}
```

## Marker Discipline (AG-5) and Ownership Guard (AG-9)

### Two-part marker check (AG-5)

The two-part marker check exists at TWO call sites:
1. **`prepare()` step 7** -- when re-staging a plugin's agents, every PREVIOUS target file (from the index) MUST pass the check. If any fails, throw `AgentForeignContentError` and refuse to proceed. The plugin's reinstall is blocked until the foreign file is removed manually.
2. **`unstage()` per-entry** -- when removing, each target file is re-checked. Failures preserve the index row + populate `failed[]` so retry surfaces the same diagnostic.

**Predicate (canonical):**
```typescript
async function isOwnedAgentFile(targetPath: string): Promise<{ok: true} | {ok: false; reason: string}> {
  // (a) basename rule
  if (!path.basename(targetPath).startsWith("pi-claude-marketplace-")) {
    return { ok: false, reason: `target filename "${path.basename(targetPath)}" does not start with "pi-claude-marketplace-"` };
  }
  // (b) body marker rule
  let contents: string;
  try { contents = await readFile(targetPath, "utf8"); }
  catch (err) {
    if ((err as NodeJS.ErrnoException).code === "ENOENT") return { ok: true };
    throw err;
  }
  if (!contents.includes("generated by pi-claude-marketplace")) {
    return { ok: false, reason: `target ${targetPath} is missing the generated marker` };
  }
  return { ok: true };
}
```

### Generating the marker for new files

The marker is emitted by `bridges/agents/frontmatter.ts::emitGeneratedAgentFile` as part of the file's body (immediately after the closing `---` of the frontmatter, in an HTML-comment provenance block):

```
---
name: pi-claude-marketplace-acme-bot
description: ...
tools: read,bash,edit
systemPromptMode: replace
inheritProjectContext: true
inheritSkills: false
---

<!--
generated by pi-claude-marketplace        <-- THE MARKER (substring match)
plugin: acme
sourceAgent: bot
sourcePath: /abs/path/to/source.md
droppedFields: (none)
droppedTools: (none)
warnings: (none)
-->

<the agent's body content>
```

**Why HTML-comment-in-body, not frontmatter:** pi-subagents' line-based frontmatter parser would treat unknown frontmatter keys as no-ops, but it would also pollute the round-trip. Putting provenance in the BODY behind an HTML comment means (a) markdown renderers hide it; (b) pi-subagents doesn't try to interpret it; (c) the file remains a substring-searchable for the marker. The leading newline before `<!--` is the deterministic separator emitted by `emitGeneratedAgentFile`.

### Foreign-content channel (AG-5 + AG-9 → `failed[]`)

The `StageAgentsCommitResult.failed[]` channel surfaces ONLY at the `unstage` boundary -- during `prepare`, foreign content at a previous target is a HARD throw (`AgentForeignContentError`) so the user knows the install/update is not safe to proceed. During `unstage`, foreign content is recorded in `failed[]` and the index row is preserved, so the next uninstall attempt (after the user manually removes the foreign file) can retry cleanly. PU-7 (PRD §5.2.2) is the user-visible spec for this: "uninstall fails loudly" -- meaning the orchestrator (Phase 5) translates `failed[]` into a thrown error.

Phase 3 design note: ON THE PREPARE PATH, foreign content is a throw (AG-5 wording: "is not overwritten"). The `failed[]` field on `StageAgentsCommitResult` is reserved for the unstage path and remains empty in commit results from `prepare → commit`.

### Cross-owner guard (AG-9 / RN-4)

Inside `prepare()` step 5: after partitioning the index by `(mp, plugin)`, build a `Map<generatedName, otherEntry>` from the others-partition. For each new converted agent, look up its generated name; if found, the agent is owned by a DIFFERENT `(mp, plugin)`.

**Exact predicate:**
```typescript
const otherNames = new Map(otherEntries.map(e => [e.generatedName, e]));
const conflicts = converted
  .map(c => ({ name: c.generatedName, owner: otherNames.get(c.generatedName) }))
  .filter((x): x is { name: string; owner: AgentIndexEntry } => x.owner !== undefined);
```

**Error message format (V1 verbatim):**
```
Refusing to stage agents for <mp>/<plugin>: "<name>" already owned by <other-mp>/<other-plugin>.
```

When N conflicts: `Refusing to stage agents for <mp>/<plugin>: "n1" already owned by <mp1>/<p1>; "n2" already owned by <mp2>/<p2>; ....`

## MCP Collision Guard and Precedence Chain

### `MCP_COLLISION_SLOTS` constant

```typescript
// bridges/mcp/slots.ts
export const MCP_COLLISION_SLOTS = (cwd: string): readonly string[] => Object.freeze([
  path.join(homedir(), ".config", "mcp", "mcp.json"),       // shared-global
  path.join(homedir(), ".pi", "agent", "mcp.json"),         // pi-user-scope
  path.join(cwd, ".mcp.json"),                              // shared-project
  path.join(cwd, ".pi", "mcp.json"),                        // pi-project-scope
]);
```

### Collision-check predicate (MC-4 / RN-5)

```typescript
const effective = await loadEffectiveServerNames(cwd);   // Map<serverName, owningPath>
for (const name of newNames) {
  if (ours.has(name)) continue;                          // self-replace within own scope -- OK

  const owningPath = effective.get(name);
  if (owningPath !== undefined && owningPath !== locations.mcpJsonPath) {
    throw new McpServerCollisionError(/* foreign collision in different slot */);
  }

  if (Object.prototype.hasOwnProperty.call(theirs, name)) {
    throw new McpServerCollisionError(/* foreign collision in same scope's mcp.json */);
  }
}
```

Three branches:
1. `ours.has(name)` -- the same `(plugin, marketplace)` previously staged this server. Allow self-replace (this is an update / re-stage).
2. `owningPath !== locations.mcpJsonPath` -- the server lives in a DIFFERENT slot (e.g., user-scope shared-global). Refuse stage.
3. `theirs[name]` exists -- the server lives in OUR scope's `mcp.json` but is NOT marked as ours (different plugin or no marker). Refuse stage.

### Precedence chain (MC-1)

`bridges/mcp/parse.ts::resolvePluginMcpServers` resolves the plugin's declared `mcpServers` map by walking three sources in order:

1. **Marketplace entry** (`entry.mcpServers`) -- if present (even if empty `{}`), use it. Malformed at this source throws.
2. **Plugin manifest** (`manifest.mcpServers`) -- if present and entry was absent, use it. Malformed throws.
3. **Standalone** `<pluginRoot>/.mcp.json` -- if file exists. Detect wrapped (`{mcpServers: {...}}`) vs unwrapped form per MC-2. Malformed JSON throws. Empty `{}` returns `{source: "none", servers: {}}`.

When all three are absent: `{source: "none", servers: {}}`.

**No fallthrough on malformed.** If `entry.mcpServers === "not-an-object"`, the function throws -- it does NOT skip to manifest. This is the explicit MC-1 contract: "malformed at the matched source MUST throw".

**Strict-false MM-7 case:** when the plugin's resolver context says `strict=false` AND entry doesn't declare `mcpServers` AND manifest/standalone do, the resolver (Phase 2) marks the plugin as non-installable with "component declarations conflict". The bridge `resolvePluginMcpServers` does NOT know about strict-mode; it's called only after the resolver decides the plugin is installable. So this conflict surfaces at the resolver tier and never reaches the bridge.

### Self-replace vs foreign refusal

| Scenario | Behavior |
|----------|----------|
| Same plugin re-stages same server name in same scope | Allowed -- self-replace (`ours.has(name)`) |
| Same plugin re-stages with different server name | Allowed -- new entry, no collision |
| Different plugin tries to register same server name in same scope (no marker on the existing entry) | Refused -- `theirs[name]` exists |
| Different plugin tries to register same server name in same scope (marker on existing entry, different (mp,plugin)) | Refused -- partitioned to `theirs`, refused at `theirs[name]` check |
| Same plugin in different scope tries to register server name owned by self in another scope | Refused -- `effective.get(name) !== locations.mcpJsonPath` |
| Server name appears in user's `.config/mcp/mcp.json` (shared-global) | Refused regardless of plugin (foreign slot) |

## Materialization Gates

Per phase success criterion 4 + AS-8 + AS-9: certain bridges MUST NOT materialize their target files when there's nothing to write.

### MCP gate (AS-8)

```typescript
// bridges/mcp/prepare.ts
if (newNames.length === 0 && ours.size === 0) {
  return { kind: "noop" };
}
```

**Predicate:** `(empty new mcpServers map) AND (no previous-ours entries in current scope's mcp.json)`.

**Effect:** `commitPreparedMcp({kind:"noop"})` returns immediately without calling `atomicWriteJson`. So if the scope's `mcp.json` doesn't exist before the install AND the plugin declares no servers, the file is NOT created.

**Why it matters:** A plugin with no MCP servers running through install must not pollute every install's mcp.json with empty `{mcpServers: {}}` entries. A scope that has only ever installed plugins-without-servers should not have an `mcp.json` at all.

### Agents gate (AS-9)

```typescript
// bridges/agents/prepare.ts
if (converted.length === 0 && previousEntries.length === 0) {
  return { kind: "noop", result: { stagedNames: [], warnings: [], failed: [] } };
}
```

**Predicate:** `(empty agents source dir / no discovered) AND (no previous-ours entries in agents-index.json for this (mp, plugin))`.

**Effect:** `commitPreparedAgents({kind:"noop"})` returns immediately. The scope's `<scopeRoot>/agents/` directory is NOT created if no agent file is staged. The `<extensionRoot>/agents-index.json` file is NOT created if no entry for this plugin exists yet.

**Note:** the gate is `converted.length === 0 && previousEntries.length === 0` -- per-(mp,plugin) ownership, not global. If OTHER plugins have agents in the index, those rows are preserved (handled by the partition in step 4 + the merge `[..._otherEntries, ..._newEntries]` in commit). The noop branch only applies when this specific (mp, plugin) has nothing to add AND nothing to remove.

### Skills / Commands gates (no REQ-ID, but for consistency)

Skills and Commands have NO Phase 3 success criterion 4 entry, but for prepare/commit symmetry the same `discovered.length === 0 && previousNames.length === 0` short-circuit applies. This avoids creating `<extensionRoot>/skills-staging/<uuid>/` (or `prompts-staging/<uuid>/`) when there's nothing to stage. The `<extensionRoot>/resources/skills/` and `<extensionRoot>/resources/prompts/` directories are still created lazily by `mkdir({recursive:true})` in commit, but ONLY when there's a rename to do -- which the noop branch skips.

## Body-Placeholder Substitution

Per D-08 + SK-4 + CM-3 + PI-10.

### Substitution function

```typescript
// shared/vars.ts
import { assertSafeName } from "../domain/name.ts";

export interface ClaudePluginVars {
  pluginRoot: string;       // absolute path to source plugin dir
  pluginData: string;       // <scopeRoot>/pi-claude-marketplace/data/<mp>/<plugin>/
}

/**
 * Substitute `${CLAUDE_PLUGIN_ROOT}` and `${CLAUDE_PLUGIN_DATA}` in plugin
 * content (skill bodies, command bodies; NOT agent bodies per D-08).
 *
 * Pure function: caller passes the resolved values, this only does string
 * replacement. Naive `replaceAll` is correct -- the literals are unambiguous
 * and don't appear in normal content.
 */
export function substituteClaudeVars(content: string, vars: ClaudePluginVars): string {
  return content
    .replaceAll("${CLAUDE_PLUGIN_ROOT}", vars.pluginRoot)
    .replaceAll("${CLAUDE_PLUGIN_DATA}", vars.pluginData);
}
```

### When substitution happens

**At PREPARE time, NOT commit time.**

- SkillsBridge: in `prepareStageSkills`, after `cp -r` of the skill dir into staging, `readFile` the staged `SKILL.md`, run `substituteClaudeVars`, `writeFile` back. Only `SKILL.md` -- other files in the skill dir (resources, scripts) are copied verbatim.
- CommandsBridge: in `prepareStageCommands`, `readFile` the source `.md` file, substitute, `writeFile` to the staged path.
- AgentsBridge: substitution applied INSIDE `convertAgent` (V1 line ~470 `substitutePluginVars(body, ...)`). D-08 explicitly says "Agents do NOT need substitution in their bodies (no AG-* requirement)" -- BUT V1 actually substitutes in agent bodies via `substitutePluginVars`. **OPEN QUESTION → planner:** D-08 is in tension with V1 behavior. PI-10 (PRD §5.2.1) says substitution applies in "skill bodies, command files, agent bodies" -- which CONTRADICTS D-08's claim that agents don't need it. RESOLVED: Phase 3 plan PRESERVES V1 behavior (substitute in agent bodies too) because PI-10 is the user-facing PRD contract; D-08's "no AG-* requirement" wording reflects that AG-* requirements don't mention it explicitly, but PI-10 requires it across all three. If planner agrees, mark D-08 as superseded by PI-10 in CONTEXT.md. **Default for RESEARCH.md: substitute in agent bodies, matching V1.**

### Resolved values

- `pluginRoot`: absolute path to the source plugin directory. From `ResolvedPluginInstallable.pluginRoot` (Phase 2 D-04). Already constrained by `assertPathInside`.
- `pluginData`: `<scopeRoot>/pi-claude-marketplace/data/<mp>/<plugin>/`. Phase 5's install orchestrator calls `locations.pluginDataDir(mp, plugin)` to compute this and creates the dir AFTER state commit (per CONTEXT.md D-08 corollary: "the data/ subdir creation lives in Phase 5's install orchestrator (after the bridge state commit), NOT in Phase 3 -- bridges only emit the substituted string; the dir doesn't need to exist for substitution to be byte-correct.").

So at Phase 3 prepare-time, the substituted body contains an absolute path that may not yet exist on disk. The path becomes valid after Phase 5 creates `<dataRoot>/<mp>/<plugin>/`. This is correct: the substitution writes a path string into the file, not a file-system reference.

## Atomicity and Rollback

### Per-bridge atomic boundary

| Bridge | Atomic Unit | Boundary Mechanism | EXDEV Risk |
|--------|-------------|--------------------|-----------|
| **Skills** | Per-skill directory rename | `fs.rename(stagedDir, targetDir)` -- POSIX atomic | NONE if same FS (staging under `<extensionRoot>/skills-staging/`, target under `<extensionRoot>/resources/skills/` -- both same FS). Cross-FS = EXDEV. |
| **Commands** | Per-file rename | `fs.rename(stagedFile, targetFile)` -- POSIX atomic | NONE if same FS (staging under `<extensionRoot>/prompts-staging/`, target under `<extensionRoot>/resources/prompts/`). |
| **Agents** | Per-file rename + atomic JSON write | `fs.rename` per file + `atomicWriteJson(agents-index.json)` | NONE if same FS (staging under `<extensionRoot>/agents-staging/`, target under `<scopeRoot>/agents/` -- BOTH on same FS as scopeRoot). |
| **MCP** | Single atomic JSON write | `atomicWriteJson(mcp.json)` -- via `write-file-atomic@^8` | NONE -- write-file-atomic creates tmp in destination dir. |

### EXDEV cross-FS failure mode

**Risk:** if a user mounts `<scopeRoot>/agents/` on a separate filesystem from `<scopeRoot>/pi-claude-marketplace/agents-staging/`, the `rename(staged, target)` will throw `EXDEV` ("invalid cross-device link").

**Mitigation in Phase 3:**
1. Per CONTEXT.md "What NOT to Use" guidance: "If staging dir and destination are on different filesystems, `rename` will EXDEV-fail. NFR-1's atomic guarantee requires same-FS." Phase 3 chooses staging dirs that are guaranteed-same-FS as their targets:
   - Skills: staging under `<extensionRoot>/skills-staging/`, target under `<extensionRoot>/resources/skills/` -- both inside `<scopeRoot>/pi-claude-marketplace/`, same FS by construction.
   - Commands: same -- staging and target both under `<extensionRoot>/`.
   - Agents: staging under `<extensionRoot>/agents-staging/`, target under `<scopeRoot>/agents/`. Both descendants of `<scopeRoot>` -- same FS by construction (a user who mounts `<scopeRoot>/agents/` on a separate FS is configuring outside the supported topology; we don't claim atomicity in that pathological case).
   - MCP: `write-file-atomic` creates tmp in destination dir -- impossible to EXDEV.
2. **No fallback to `cp + unlink`.** Per CONTEXT.md: "for any cross-FS ops, fall back to copy-then-fsync-then-unlink, never claim atomicity." Phase 3 does NOT implement a copy-fallback; it relies on the same-FS topology.

### Rollback story per bridge

| Bridge | If commit fails after rename N of M | Rollback strategy |
|--------|--------------------------------------|-------------------|
| Skills | Skills 1..N renamed; N+1..M still in staging | `unstage` is idempotent (ENOENT-tolerant). Phase 5's ledger calls `unstage` as the `undo`; it removes ALL `previousNames + newlyStagedNames` from target dir (whichever subset exists), then leaves the staging UUID dir orphaned (`abort` cleans it on a separate path; here the unstage doesn't touch it). |
| Commands | Same as skills, per-file granularity | Same idempotent `unstage`. |
| Agents | Files 1..N renamed; the rest still staged; index.json NOT yet written (saveAgentIndex is the LAST step, after all renames) | If failure between renames and saveAgentIndex: on-disk has new files but index still describes OLD state. The next call to `unstagePluginAgents` reads the OLD index, attempts to remove the OLD entries (which were already removed in step 6), tolerates ENOENT, succeeds. The new files become orphaned. **The next `prepareStagePluginAgents` for this (mp, plugin) finds zero `previousEntries` (because the index was never updated) and the orphans pass `isOwnedAgentFile` (they have the marker). The new prepare's converted agents have the SAME generated name as the orphans → `rename(staged, target)` overwrites the orphan. Self-heal.** |
| MCP | `atomicWriteJson` is single-shot; either fully written or not. No partial state. | Failure during write: file may be left as the OLD content (write-file-atomic's tmp was never renamed) OR as a dangling tmp in the dest dir (cleanup on next process boot via signal-exit hooks). Either way, the in-memory rollback is irrelevant -- nothing committed. |

### Phase 5 ledger composition (forward-look per D-02)

Per D-02 each bridge maps to one literal `Phase<C>` in the install ledger. The 4-phase install order from PRD §5.7 (skills → commands → agents → MCP → state) becomes a 5-element literal array. When phase N fails, `runPhases` runs phases 1..N-1 `undo` in reverse order. Each bridge's `unstage` is the `undo`. Concretely:

```
[skills, commands, agents, mcp, state]

Failure at "agents":
  → runPhases unwinds: state.undo() (no-op), mcp.undo() (no-op or noop), agents.undo() (idempotent unstage), commands.undo() (idempotent unstage), skills.undo() (idempotent unstage)
  → bridge prepares (skillsPrep, commandsPrep, ...) live OUTSIDE the ledger;
    the ledger entry's `do` calls `commit(prep)`. If commit fails partway,
    the bridge's own retry-self-heal (above) is the LAST resort; the ledger
    just calls undo on previously-committed phases.
```

Phase 3 ships the bridges as primitives; Phase 5 wires this composition.

## Pitfalls and Anti-patterns

### What V1 got right (MUST NOT regress)

1. **Index partitioning by `(marketplace, plugin)` tuple.** Phase 3 MUST preserve the partition logic in `prepareStagePluginAgents` step 4. Re-staging plugin X in marketplace M1 MUST leave plugin X in marketplace M2's rows untouched. Single-mp partition was a V1 bug fixed before V1 shipped -- do not regress.
2. **Two-part marker check (basename AND body).** Both are required; either failing is a refusal. V1's `isSafeToTouch` enforces both; Phase 3's `isOwnedAgentFile` MUST do the same.
3. **`saveAgentIndex` is the LAST step in commit, not the first.** If saveAgentIndex fails, file-system has new files but index still describes OLD state. The next `unstage` cleans old (now-missing) files via ENOENT-tolerance, and the next `prepare` overwrites the orphans because they have the marker. Self-heals on retry.
4. **`MODEL_MAP` and `TOOL_MAP` are user contract.** A plugin author wrote `model: sonnet` knowing it maps to `anthropic/claude-sonnet-4-6`. Any change here breaks downstream agents. Snapshot test these maps to detect accidental mutation.
5. **Source hash over RAW BYTES, not utf8 text.** V1's `discoverPluginAgents` does `const bytes = await readFile(sourcePath); const sourceHash = createHash("sha256").update(bytes).digest("hex");`. This is BOM/line-ending tolerant -- different OS line endings on the same source produce the same hash. PI-7 is at the plugin-version layer (Phase 5), not agents bridge, but the agents-bridge sourceHash must match.
6. **`_piClaudeMarketplace` marker is per-server, not per-doc.** Each staged server entry has its own marker; the doc itself doesn't carry one. This lets two plugins coexist in the same `mcp.json` and unstage independently.
7. **First-declarer-wins in `loadEffectiveServerNames`.** Read order is fixed (shared-global → pi-user → shared-project → pi-project). Changing the order breaks MC-4 collision behavior.
8. **`emitYamlScalar` quote-flip handles the double-quote-double-quote and single-quote-single-quote cases.** A description like `"hello"` or `'world'` gets the OPPOSING quote wrapper so pi-subagents' naive parser doesn't strip both ends.
9. **`sanitizeProvenance` `-->` → `--&gt;`.** A source path containing `-->` would terminate the HTML comment early; sanitization prevents that.

### What V1 got wrong (or under-spec) that the successor MUST fix

1. **Skills + commands bridges share a single `resource/stage.ts`.** Phase 3 splits them into `bridges/skills/` and `bridges/commands/` per the 9-folder layout. **Risk:** keeping them combined would force Phase 5 to call ONE function for both, making bridge-as-Phase composition (D-02) impossible.
2. **`agents-index.json` schema validation is hand-rolled.** V1's `validateAgentIndexEntry` is duplicate-implementation of TypeBox. Phase 3 hoists to `persistence/agents-index-schema.ts` with `Compile(SCHEMA)` JIT validator. Same correctness, less code, faster validation.
3. **No typed errors for foreign content / cross-owner / mcp collision.** V1 throws plain `Error`. Phase 3 introduces `AgentForeignContentError`, `AgentOwnershipConflictError`, `McpServerCollisionError` so test-side `instanceof` checks work and Phase 5 cross-bridge aggregation can `instanceof`-distinguish.
4. **No bridge-local prepare/commit for skills + commands.** V1's orchestrator owns the staging tmp dir. Phase 3's bridges own their own staging dirs (per D-04 atomic-apply granularity).
5. **`MCP_COLLISION_SLOTS` inlined.** V1 has the four paths inlined in `loadEffectiveServerNames`. Phase 3 hoists to a named constant snapshot-tested at `tests/bridges/mcp/slots.test.ts` per CONTEXT.md "Specific Ideas."

### Easy mistakes a planner could make

1. **Putting `cwd` in `ScopedLocations`.** `cwd` is per-process state; `ScopedLocations` is per-scope identity. The cleanest path: `StageMcpInput` includes `cwd: string` separately. Don't touch `locations.ts` to add a `cwd` field.
2. **Calling `notifyWarning` from inside a bridge.** Bridges are pure primitives -- they throw or return `CommitResult` with `warnings[]`. Phase 4/5 orchestrators surface warnings via notify. Adding a notify call inside a bridge breaks IL-2 and the unit-test isolation (tests would need to mock notify).
3. **Conflating `abort` and `unstage`.** `abort(prepared)` is for "the prepare succeeded but I decided not to commit" (cleanup staging dir). `unstage(target)` is for "this plugin's previously-committed resources should be removed" (reverse a successful commit). Calling `abort` after a successful commit is a no-op (staging is already cleaned); calling `unstage` after `abort` is wrong (there's nothing to unstage, the prepare never committed). Phase 5's ledger uses `unstage` as `Phase.undo`, NOT `abort`.
4. **Putting `previousNames` discovery in the bridge.** Bridges shouldn't read `state.json` -- they accept `previousNames: readonly string[]` as input. Phase 5's orchestrator reads state and passes the names. Otherwise the bridge becomes coupled to `persistence/state-io.ts` and the unit-test setup gets ugly.
5. **Using `fs.cp({recursive:true})` to commit a staging tree.** `fs.cp` is OK for prepare (copying source skill dir into staging tmp), but the commit-side rename of staging-to-target MUST be `fs.rename`, not `fs.cp + fs.rm`. `fs.rename` is atomic on same FS; `cp+rm` is not.
6. **Forgetting `assertPathInside` on staged paths.** Every name-derived path -- including the `<staging>/<generatedName>` path -- must go through `assertPathInside` to defend against `generatedName === "../escape"`. The Phase 2 generators (`generatedSkillName` etc.) already call `assertSafeName`, but the `path.join` result must still be `assertPathInside`'d (defense in depth).
7. **Trusting `fs.cp` on symlinks.** A skill source dir might contain symlinks (a malicious or careless author). `fs.cp({recursive: true, dereference: false})` is the default and follows symlinks -- which COULD escape. Phase 3 SHOULD pass `dereference: false, verbatimSymlinks: true` so symlinks are copied as symlinks, NOT resolved. **OPEN QUESTION → planner:** how does Phase 3 handle symlinks inside a skill dir? Phase 1 D-14 refuses ALL symlinks via `assertPathInside` walking parents -- but that's at `assertPathInside` time on the destination, not on the COPIED source content. RESOLVED: walk the source skill dir with `lstat` before `cp`, refuse if any inner path is a symlink. (V1 doesn't do this -- gap in V1.)
8. **Renaming `agents-index.json` field from `agents` to `entries`.** V1 stored entries under field `agents`. Renaming breaks every existing user's wire shape. Phase 3 MUST preserve the V1 field name `agents` in the schema; rename only at hypothetical schemaVersion 2.
9. **Using `process.cwd()` inside the bridge.** V1's `loadEffectiveServerNames(cwd)` takes cwd as a param. Phase 3 MUST do the same -- pass cwd into the bridge via input. Calling `process.cwd()` inside a bridge breaks unit testability (tests would need to chdir, breaking parallelism).
10. **Computing `pluginDataDir` differently in the bridge vs. orchestrator.** The bridge's substituted body has the dataDir path embedded as a literal string. If Phase 3's bridge computes it as `<extensionRoot>/data/<mp>/<plugin>/` and Phase 5's orchestrator computes it as `<scopeRoot>/pi-claude-marketplace/data/<mp>/<plugin>/` -- well, those are identical strings -- but if a future refactor moves dataDir, BOTH must update or substitution breaks. **Mitigation:** Phase 3's bridge accepts `pluginDataDir: string` as input -- the orchestrator computes it once via `locations.pluginDataDir(mp, plugin)` and passes it down.

### Pattern 1: Discriminated `Prepared<bridge>` handle

**What:** Return type from `prepare()` carrying either a noop sentinel or a fully-staged tmp tree + computed deltas.

**When to use:** Every bridge's prepare/commit pair.

**Example:**
```typescript
// Source: bridges/agents/types.ts
export type PreparedAgentsStaging = PreparedAgentsNoop | PreparedAgentsStaged;

export interface PreparedAgentsNoop {
  readonly kind: "noop";
  readonly result: StageAgentsCommitResult;
}

export interface PreparedAgentsStaged {
  readonly kind: "staged";
  readonly locations: ScopedLocations;
  readonly stagingDir: string;
  readonly result: StageAgentsCommitResult;
  readonly _previousEntries: readonly AgentIndexEntry[];
  readonly _otherEntries: readonly AgentIndexEntry[];
  readonly _newEntries: readonly AgentIndexEntry[];
  readonly _stagedFilePaths: readonly { from: string; to: string }[];
}
```

### Pattern 2: Per-bridge typed error classes

**What:** Each bridge defines its own `Error` subclasses for specific refusal modes. Subclasses inherit from `PathContainmentError` when the refusal is path-related.

**When to use:** When the bridge needs `instanceof`-distinguishable error types.

**Example:**
```typescript
// Source: bridges/agents/errors.ts
import { PathContainmentError } from "../../shared/path-safety.ts";

export class AgentForeignContentError extends PathContainmentError {
  constructor(message: string, public readonly targetPath: string) {
    super("", "", "agent target");  // base class fields not meaningful here
    this.name = "AgentForeignContentError";
    this.message = message;
  }
}

export class AgentOwnershipConflictError extends Error {
  constructor(
    message: string,
    public readonly conflicts: readonly { name: string; marketplace: string; plugin: string }[],
  ) {
    super(message);
    this.name = "AgentOwnershipConflictError";
  }
}
```

### Pattern 3: Idempotent `unstage` with ENOENT-tolerance

**What:** `unstage` removes target files/dirs, treating ENOENT as success. Re-running on already-cleaned state returns empty `removedNames` without throwing.

**When to use:** Every bridge's unstage. Phase 5's ledger calls it as `Phase.undo`; if the rollback runs after a partial commit, half the files may already be gone -- must not throw.

### Anti-Patterns to Avoid

- **Cross-bridge imports.** `bridges/skills/` MUST NOT import from `bridges/agents/`. If shared logic emerges, hoist to `domain/` or `shared/`.
- **Bridge calling `notify*`.** Bridges throw or return CommitResult; orchestrators notify.
- **Bridge reading `state.json`.** Bridges accept inputs computed by Phase 5 orchestrators (which read state).
- **`fs.cp` for commit-side tree move.** Use `fs.rename` for atomic commit.
- **`process.cwd()` inside a bridge.** Pass `cwd` as input.
- **`fs.writeFileSync` anywhere.** Async only.

## Common Pitfalls

### Pitfall 1: Bridge importing transaction/

**What goes wrong:** A bridge calls `withStateGuard` directly because "it sounds like a good idea." This violates D-11 import boundaries and couples bridges to the state-mutation layer.

**Why it happens:** The bridge author needs to ensure no concurrent re-stage; `withStateGuard` looks like the right primitive.

**How to avoid:** Bridges accept inputs (`previousNames`, `cwd`, etc.) and assume the caller has already serialized via `withStateGuard`. Phase 5's ledger composition does this.

**Warning signs:** ESLint `import-x/no-restricted-paths` violation (Phase 1 D-11 already enforces this).

### Pitfall 2: `prepare()` doing partial disk writes that aren't tracked for rollback

**What goes wrong:** A bridge writes some files into staging, throws, but the staging dir is never cleaned (because the throw happens BEFORE the staging dir is recorded in the `Prepared<bridge>` handle).

**Why it happens:** The author writes the cleanup-on-error wrapper too late in the function.

**How to avoid:** Wrap every disk write inside `prepare` with a try/catch that calls `cleanupStaging(stagingDir)` and uses `appendLeakToError` to surface the leak. V1 does this correctly:

```typescript
try {
  for (const c of converted) { /* writeFile */ }
} catch (err) {
  throw appendLeakToError(err, await cleanupAgentsStaging(stagingDir));
}
```

Carry forward.

**Warning signs:** Test fails when prepare throws halfway -- staging UUID dir is left on disk after the test.

### Pitfall 3: Foreign-content check happening AFTER the disk write

**What goes wrong:** A bridge writes new staged files first, then checks if the previous target file is foreign. If foreign, the bridge throws -- but the staged files are now orphaned in the staging dir.

**Why it happens:** Author orders steps for "obvious" workflow (write new, then check if old can be replaced) instead of "check first, then write."

**How to avoid:** Step 4 in V1 `prepareStagePluginAgents` -- "Safety-check each previous target before any disk write." Phase 3 MUST preserve this ordering.

**Warning signs:** The test for foreign-content cases passes (throw happens) but a subsequent test sees a staging-tmp leak.

### Pitfall 4: `agents-index.json` saved BEFORE renames complete

**What goes wrong:** Bridge saves the new index first, then renames staged files. If renames fail, index says "I have these agents" but disk doesn't.

**Why it happens:** Author thinks "save the source of truth first."

**How to avoid:** V1 ordering -- renames FIRST, then `saveAgentIndex`. If saveAgentIndex fails, on-disk has new files but index still describes OLD entries; the next `unstage` cleans them via marker check + retry.

**Warning signs:** Test where saveAgentIndex throws shows disk in inconsistent state that retry doesn't fix.

### Pitfall 5: `MCP_COLLISION_SLOTS` order changed

**What goes wrong:** Someone reorders the four slots "for clarity" and breaks first-declarer-wins precedence in `loadEffectiveServerNames`.

**Why it happens:** Slot order looks arbitrary; someone alphabetizes.

**How to avoid:** Snapshot test in `tests/bridges/mcp/slots.test.ts` pins the four paths in order. Any reorder fails the test.

**Warning signs:** A previously-passing collision test starts failing because the user's `.config/mcp/mcp.json` is no longer "first."

### Pitfall 6: Substitution of `${CLAUDE_PLUGIN_DATA}` to a non-existent path

**What goes wrong:** Phase 3 bridge substitutes `${CLAUDE_PLUGIN_DATA}` to `<dataRoot>/<mp>/<plugin>/`, but the dir hasn't been created yet (Phase 5 creates it post-state-commit). At runtime when Pi reads the SKILL.md and resolves the path, the dir doesn't exist -- skill features that reference data files break.

**Why it happens:** The substitution writes a path string; runtime uses the string assuming the path exists.

**How to avoid:** Phase 5's install orchestrator MUST create `<dataRoot>/<mp>/<plugin>/` AFTER the bridge state commit but BEFORE the user runs `/reload`. Document this dependency in the Phase 3 RESEARCH (this section) AND in the bridge READMEs.

**Warning signs:** Skill body reads a file from `${CLAUDE_PLUGIN_DATA}/...` and fails at first run.

## Open Questions (RESOLVED)

1. **`agents-index.json` field name compatibility (A1).**
   - What we know: V1 stores entries under `agents:`. Phase 3 D-07 introduces a TypeBox schema.
   - What's unclear: Does the schema preserve `agents:` field name or rename to `entries:`?
   - RESOLVED: PRESERVE `agents:` for byte-compat. Rename only at hypothetical schemaVersion 2. Planner verifies in 03-01-PLAN.

2. **Variable substitution in agent bodies (A2; D-08 vs PI-10).**
   - What we know: V1 substitutes in agent bodies. PRD §5.2.1 PI-10 says substitution applies in skill bodies, command files, AND agent bodies. CONTEXT.md D-08 says "Agents do NOT need substitution".
   - What's unclear: Which prevails -- PRD PI-10 or CONTEXT.md D-08?
   - RESOLVED: PI-10 prevails (PRD is the spec; D-08's wording reflects that AG-* requirements don't list it but PI-10 does). Phase 3 plan substitutes in agent bodies -- matches V1.

3. **Symlink handling inside skill source dirs (A3).**
   - What we know: Phase 1 D-14 refuses ALL symlinks via `assertPathInside`. But that protects the BRIDGE'S target paths, not the source dir's CONTENTS being copied.
   - What's unclear: Should `bridges/skills/prepare.ts` walk the source skill dir with `lstat` and refuse if any inner path is a symlink, before `cp`?
   - RESOLVED: YES -- walk + refuse. Add to Phase 3 plan as a hardening over V1 (V1 has this gap).

4. **Skills directory hardening: dotfiles, control-char names.**
   - What we know: V1 calls `assertSafeName(entry.name, ...)` which throws on path separators + control chars but DOES allow dotfiles (e.g., `.git`).
   - What's unclear: Should `bridges/skills/discover.ts` skip directories starting with `.` (dotfiles)?
   - RESOLVED: skip dotfiles by convention (a `.git` subdir under `skills/` is probably an artifact of a misplaced `.gitignore`, not a real skill). Phase 3 plan adds this.

5. **`bridges/mcp/parse.ts` location vs Phase 2's `domain/components/mcp.ts`.**
   - What we know: CONTEXT.md "Phase 2 carry-forward" mentions `domain/components/mcp.ts` -- which suggests Phase 2 may already ship the schema/validator. V1's `mcp/parse.ts` does both shape validation AND precedence chain resolution.
   - What's unclear: Does `parseMcpServers` (shape) live in `domain/components/mcp.ts` (Phase 2) and `resolvePluginMcpServers` (precedence) in `bridges/mcp/parse.ts` (Phase 3)? Or both in one file?
   - RESOLVED: split. Schema-validation primitive in Phase 2 `domain/components/mcp.ts`; precedence-chain orchestration in Phase 3 `bridges/mcp/parse.ts`. Planner verifies Phase 2's actual shipped surface in 03-01-PLAN.

6. **Phase 5 cross-plugin guard (PI-6) ordering vs bridge prepare.**
   - What we know: D-05 says cross-bridge name guard runs in Phase 5 BEFORE any bridge's prepare.
   - What's unclear: How does Phase 5 know what generated names a bridge will produce without running prepare? Answer: Phase 5 calls `discoverPluginSkills` / `discoverPluginCommands` / `discoverPluginAgents` to get the generated-name list (which is pure, no disk writes), runs PI-6 cross-check, THEN calls prepare on each bridge.
   - RESOLVED: Phase 3 EXPORTS the discovery functions as standalone (not just internal to prepare) so Phase 5 can call them without prepare's side effects. RESEARCH.md treats `discoverPluginSkills` etc. as already public; planner confirms.
