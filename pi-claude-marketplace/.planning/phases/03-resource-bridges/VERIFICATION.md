## Verification Check 4: Integration Capstone -- Four-Bridge Composition

**Check:** Does the integration capstone actually compose all four bridges end-to-end, or is it bridge-isolated tests in a shared file?

**Finding: VERIFIED -- genuine end-to-end composition**

`tests/bridges/integration.test.ts` uses a single shared `scopeRoot` (one `mkdtemp` in `before`) and runs all four bridges sequentially against it in the same `describe("integration: full-plugin staging")` block. Each bridge's test builds on the prior state:

1. Skills test: commits skills to `scopeRoot` -- verifies SK-2 elision, SK-3 frontmatter rewrite, SK-4 var substitution
2. Commands test: commits commands to the same `scopeRoot` -- verifies CM-2 elision, CM-3 var substitution
3. Agents test: commits agents to the same `scopeRoot`, reads `agents-index.json` -- verifies AG-1 basename, AG-2/3 index partitioning, AG-5 marker
4. MCP test: commits MCP to the same `scopeRoot`, reads written `mcp.json` -- verifies MC-1 precedence, MC-5 marker, MC-6 recorded

The idempotency test (fifth in the suite) re-stages skills on the same `scopeRoot` verifying the re-stage path works.

`integration-foreign-content.test.ts` and `integration-materialization-gate.test.ts` are separate files (by D-01 design -- each bridge is siloed) and exercise orthogonal concerns, not duplicates of the main integration. The materialization-gate test explicitly verifies cross-bridge isolation (MCP-only stage does not materialize `agentsIndexPath`, `agentsDir`, `skillsTargetDir`, `promptsTargetDir`).
