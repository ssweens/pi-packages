<objective>
Land the TypeBox manifest schemas + JIT-compiled validators for `marketplace.json` and `plugin.json`. Per CONTEXT.md D-05, schemas live in `domain/manifest.ts` (top-level marketplace) + `domain/components/plugin.ts` (plugin-entry + plugin-manifest) + `domain/components/mcp.ts` (mcpServers map). Per CONTEXT.md D-07, JIT compilation runs at module load -- compiled validators are just additional exports. **Three critical TypeBox 1.x API corrections vs. CONTEXT.md wording (per RESEARCH.md Pitfalls 1-3):**
1. `Type.Union([...], { discriminator: 'kind' })` does NOT exist in TypeBox 1.x -- use literal-tagged variants (`Type.Object` with `Type.Literal('tag')`); TypeScript narrowing is identical (Pitfall 1)
2. `Type.Recursive` was REMOVED in TypeBox 1.0 -- use `Type.Cyclic({Plugin: ...}, 'Plugin')` or `Type.Module({...})` (Pitfall 2)
3. `import { TypeCompiler } from '@sinclair/typebox/compiler'` is the 0.34 path -- 1.x uses `import { Compile } from 'typebox/compile'` (Pitfall 3)

Per RESEARCH.md Open Question 1, only `plugin.ts` and `mcp.ts` component schemas land in Phase 2. Skill/command/agent schemas are runtime-discovered (line-parsed YAML frontmatter, not `marketplace.json`/`plugin.json` fields) and land in Phase 3 with their bridges.

Purpose: `domain/resolver.ts` (Plan 02-05) consumes these schemas to validate manifest JSON before classification. `persistence/state-io.ts` (Plan 02-04) does NOT consume manifest schemas (state.json has its own schema). Phase 3 MCP bridge consumes `MCP_SERVERS_SCHEMA`.
Output: 3 schema files + 1 test file; all JIT-compiled at module load.
